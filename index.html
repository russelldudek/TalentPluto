The decision architecture

Four dispositions. One explicit reason.

CONFIGURE

The shared product already solves the need.

Bound the setup, name the owner, and measure time-to-value.

PROTOTYPE

An important uncertainty still needs evidence.

Build the smallest test that can reduce it without creating hidden commitment.

PRODUCTIZE

The need repeats and the economics support reuse.

Convert the proven behavior into a reusable primitive, playbook, or product surface.

DECLINE

The work has weak leverage or unsafe ambiguity.

Protect the roadmap by making the boundary explicit rather than allowing a silent exception.

First 90 days

Read the system. Prove one plan. Install the learning return.

01 · Days 1–15

Read the account system

Shadow active accounts, founder decisions, onboarding, customization, prototypes, and engineering handoffs. Establish where context is captured—and where it disappears.

02 · Days 16–30

Standardize the query plan

Introduce the shared request brief, proof standard, product-decision record, and a weekly client-to-product review.

03 · Days 31–60

Prove one complete path

Take one representative request from customer language through a production-minded prototype, explicit evaluation, handoff, and product disposition.

04 · Days 61–90

Make the learning reusable

Convert the successful path into onboarding and handoff patterns, then prioritize the roadmap by client value, reuse, confidence, and support burden.

Executive questions

Questions that reveal whether client work is compounding.

  1. Where does enterprise customization create the most product learning today—and where does it create the most hidden rework?
  2. What evidence must exist before a prototype becomes an engineering commitment?
  3. Which customer outcomes are repeated enough to become reusable product primitives?
  4. How is founder roadmap judgment captured so the next account benefits from it?
  5. What would make this hire unquestionably valuable by day 90?
