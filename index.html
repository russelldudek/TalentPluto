First 90 days

Earn context. Prove one loop. Install the learning cadence.

Days 1–15

Read the system

Shadow client conversations, map the product surface, inspect handoffs, and learn which decisions currently lack context.

Days 16–30

Instrument the requests

Introduce a lightweight request brief capturing outcome, context, constraint, smallest proof, reuse signal, and decision owner.

Days 31–60

Prove one loop

Take one real request from discovery through a bounded prototype and an explicit configure, prototype, productize, or decline disposition.

Days 61–90

Make learning repeatable

Install the weekly client-to-product review, scorecard, handoff contract, and client communication pattern.

Executive discovery

Questions that reveal whether the operating loop is healthy.

  1. Where does important customer context most often disappear between discovery, implementation, and product?
  2. Which enterprise requests create the most value—and which quietly create the most recurring support burden?
  3. What evidence currently moves a request from account-specific work into the shared roadmap?
  4. Where should this role have unilateral decision rights, and where must product or engineering retain authority?
  5. What would make the first 90 days feel unusually successful to the founder, product lead, and customer-facing team?
